## 第一部分：项目概述

### 1.1 项目定位

灵码是一个面向数据结构与算法学习的交互式 Web 平台，将可视化演示、结构化教程、在线判题、AI 智能出题、AI 代码复审和提示词工程训练整合在同一个应用中。

- 在线地址：`https://lingma.cornna.xyz`
- 开源协议：MIT

### 1.2 解决的问题

传统数据结构教学存在三个痛点：
1. 算法执行过程抽象，学生难以直观理解指针移动、节点交换等操作
2. 编程练习反馈周期长，写完代码要手动编译运行、对比输出
3. 缺乏个性化学习路径，不同水平的学生使用相同的教材和习题

灵码通过交互式动画、在线即时判题、AI 自适应出题和学习追踪来解决这些问题。

### 1.3 目标用户

- 计算机专业本科生（数据结构课程学习）
- 考研复习学生（408 数据结构部分）
- 编程初学者（算法入门）
- 对提示词工程感兴趣的开发者（Vibe Coding 模块）

### 3.2 算法可视化模块

5 类可视化组件，覆盖 16 个算法：

**链表可视化 (LinkedListVisualization)**
- 支持操作：头插、尾插、指定位置插入、删除、查找
- 实现：生成分步动画序列，每步包含节点状态（normal/new/highlight/found）和箭头状态（normal/new/breaking/highlight）
- 代码同步：右侧显示 C++/Java/Python 三种语言代码，动画每一步高亮对应代码行
- 交互：速度控制、暂停/播放、单步前进/后退

**栈可视化 (StackVisualization)**
- 支持操作：push、pop
- 视觉效果：元素状态着色（pushing/popping/top/normal），自动播放间隔 1200ms
- 代码同步：C++ 代码行高亮

**队列可视化 (QueueVisualization)**
- 实现循环队列逻辑：`(rear+1) % MAXSIZE` 判满
- 可视化 front/rear 指针位置
- 支持 C++/Java/Python 三语言代码同步

**排序可视化 (SortVisualization)**
- 7 种算法：冒泡、选择、插入、希尔、归并、快速、堆排序
- 柱状图动画：状态着色（comparing/swapping/sorted/pivot/merge）
- 每种算法显示时间复杂度、空间复杂度、稳定性信息
- 支持 C++/Java/Python 三语言代码，每步高亮对应行

**二叉树可视化 (TreeVisualization)**
- 4 种遍历：先序、中序、后序、层序
- SVG 渲染固定位置的树节点
- 动画展示 visiting/visited 状态，逐步构建遍历结果数组
- 支持 C++/Java/Python 三语言代码同步

|------|---------|------|
| 绪论 | 4 | 数据结构定义、算法分析、时间/空间复杂度 |
| 线性表 | 6 | 顺序表、单链表、双链表、循环链表、栈、队列 |
| 树 | 6 | 基本概念、二叉树、遍历、BST、AVL、哈夫曼 |

### 3.4 编程练习模块

Practice 页面是最复杂的页面（600+ 行），分三个标签页：

**预设题库 (Preset)**
- 题库来源：7 个题库模块并行懒加载（foundations、patterns、structures、basics、classics、digitalLogic、moreArray2D），自动去重
- 分类体系：约 20 个分类，归入 foundations/structures/patterns 三大组
- 筛选功能：分类、难度、题型（编程/填空）、完成状态，支持模糊搜索
- 智能推荐：基于用户技能水平和学习历史，每日生成个性化推荐题目
- 练习路径：4 个阶段（warmup 热身 → solid 巩固 → upgrade 进阶 → sprint 冲刺）
- 复习队列：自动收集做错的题目供复习
- 多语言支持：C/C++/Java/C#/Python，每种语言提供外部资源链接（Exercism 题库）

**AI 智能出题 (AI Generator)**
- 用户选择知识点和难度，AI 实时生成题目
- 支持两种题型：编程题（含多语言模板+解答+测试用例+提示+解析）和填空题
- AI 返回的 JSON 经过严格的字段校验和归一化处理（处理各种 key 别名如 `c++`→`cpp`、`python3`→`python`）
- 支持流式生成（SSE）

**Vibe Coding Lab**（详见 3.6）

**编程题组件 (CodingExercise)**
- 内嵌代码编辑器，支持多语言切换
- 提交后调用判题服务，逐测试用例运行
- 结果展示：每个用例的通过/失败、实际输出 vs 期望输出、运行时间
- Checkpoint 机制：测试用例分组（sample/basic/boundary/stress），按组统计通过率和得分
- AI 代码复审：判题完成后自动触发，从 correctness(0-40)、boundaryRobustness(0-20)、complexityAndPerformance(0-20)、codeQualityAndReadability(0-20) 四个维度打分，给出改进建议

### 3.5 AI 思维导图

MindMap 页面（1000+ 行），功能：

- **三种输入源**：主题关键词、URL（通过后端抓取提取文本）、文件上传
- **两种生成模式**：full（完整生成含笔记）、explore（仅结构）
- **画布交互**：指针拖拽平移、滚轮缩放（0.35x ~ 2.4x）、触控板支持
- **节点操作**：添加子节点、添加兄弟节点、编辑标题/笔记、上移/下移、删除、展开/折叠
- **AI 扩展**：选中节点后调用 AI 展开补充内容
- **布局算法**：自定义树形布局，计算子树高度、分配位置、贝塞尔曲线连线，支持 CJK 字符文本换行
- **持久化**：localStorage 本地存储 + 可选的服务器云端同步（通过 `VITE_ENABLE_REMOTE_MINDMAP_SYNC` 开关）
- **导出**：JSON 格式导出/导入
- **个性化**：根据用户学习历史构建上下文，AI 生成的导图会考虑用户已学内容

### 3.6 Vibe Coding Lab（提示词工程训练）

**Prompt Arena 模式**
- 5 个赛道：frontend（前端开发）、backend（后端开发）、debugging（调试）、refactoring（重构）、review（代码审查）
- 3 个难度：beginner（入门）、intermediate（进阶）、advanced（困难）
- 流程：AI 生成场景化挑战 → 用户撰写提示词 → AI 四维度评分
- 4 个评分维度：goal_clarity（目标清晰度）、boundary_constraints（边界约束）、verification_design（验证设计）、output_format（输出格式）
- 评估结果包含：总分、各维度分数、优势列表、不足列表、改写示例、下一次推荐难度
- 用户画像：基于最近 5 次提交计算，包含推荐赛道、推荐难度、最弱维度、各赛道平均分

**Frontend Live Build 模式**
- 多轮对话式交互：用户描述需求 → AI 生成 HTML/CSS/JS → iframe 沙箱实时预览
- 支持迭代改进：每轮生成后提供建议 chips，用户可继续对话优化
- 会话管理：创建/列表/详情/下载，最多保存 12 个会话
- 产物结构：每轮生成包含 title、summary、html、css、js、mergedHtml、nextSuggestions

**方法论文档 (Methodology)**
- 10 个方法论单元，每个包含：概述、知识点（含详解/示例/技巧）、实践提示、常见陷阱
- 单元主题：专业闭环、需求拆解、上下文管理、边界约束、验证设计、输出格式、调试策略、重构方法、代码审查、持续改进
- 桌面端侧边栏 TOC + 移动端底部浮动按钮 TOC，IntersectionObserver 滚动高亮

### 3.7 用户系统与学习追踪

**认证**
- Cookie-based 会话认证（`lingma_session`，httponly，samesite=lax）
- 注册收集：用户名、邮箱、密码、技能水平、目标编程语言
- 4 个技能水平：beginner / foundation / intermediate / advanced
- 5 种目标语言：C / C++ / Java / C# / Python
- 会话有效期 30 天，过期自动清理

**进度追踪**
- 已完成课程列表 + 已完成练习列表
- 学习历史记录（课程ID、标题、分类、完成时间、时长）
- 练习历史记录（题目ID、标题、分类、得分、通过率、判定结果、运行时间、checkpoint 通过数）
- 连续学习天数（streak）计算
- 总学习时长统计
- 数据持久化：localStorage（按用户 ID 命名空间隔离）+ 服务端认证

**个性化推荐引擎 (userPersonalization.ts)**
- 每个技能水平定义：默认难度、推荐练习赛道、练习分类范围、AI 出题分类、AI prompt 消息
- `buildDailyRecommendation()` 算法：
  1. 根据最近练习表现调整有效技能水平（连续做对升级，连续做错降级）
  2. 判断学习状态：onboarding（新用户）/ review（有错题需复习）/ steady（稳定练习）/ challenge（挑战进阶）
  3. 基于日期哈希确定性选择每日焦点分类（同一天推荐相同分类）
  4. 推荐具体题目

### 3.8 国际化 (i18n)

- 支持 `zh-CN`（默认）和 `en-US`
- 12 个翻译模块并行懒加载：common、content、data、exerciseTitles、generated、lesson、page、practice、service、tutorialComponent、vibeCoding、visualization
- 三层翻译策略：精确匹配 → 正则模式匹配 → 片段替换（最长子串优先，处理部分 CJK 文本）
- `LocalizationBridge` 组件：使用 `MutationObserver` 监听 DOM 变化，自动翻译新增/修改的文本节点和可翻译属性（placeholder、title、aria-label），跳过 `<script>`/`<style>`/`<pre>`/`<code>`/`<textarea>`
- 服务层翻译：`runtimeLocale.ts` 提供 React 组件外的翻译能力（从 localStorage 读取语言设置）

### 3.9 特色 UI 组件

**PixelCat 像素猫吉祥物**
- 可拖拽的 SVG 像素猫，随机行为：睡觉（带动画 "z"）、眨眼、弹出随机语音气泡
- 尾巴摇摆动画，低动效模式下简化

**InteractiveGrid 粒子网格背景**
- Canvas 全屏粒子网格，粒子受鼠标排斥
- 粒子获得"能量"后颜色从克莱因蓝渐变到松黄
- 高能粒子之间绘制连接线
- 主题感知着色

**全局极光背景**
- 克莱因蓝 + 松黄径向渐变
- Framer Motion 浮动光球动画（20s/25s 循环）
- 低动效模式下退化为静态渐变

### 4.3 AI 集成细节

**多供应商适配**
- 支持供应商：gmn（默认）、openai、deepseek、zhipu（智谱）、modelscope、custom
- 统一使用 Responses API 协议，旧版 chat/completions 格式在后端自动转换
- 默认模型：`gpt-5.4`，客户端可覆盖
- API 密钥仅存储在服务器端 `deploy/.env`，前端不暴露

**流式传输 (SSE)**
- 所有 AI 相关端点都有流式变体
- SSE 事件类型：`preview`（进度文本）、`final`（完成载荷）、`error`（错误）
- Nginx 配置：流式端点关闭 `proxy_buffering`，超时 300 秒
- 前端 `streamingSse.ts` 通用 SSE 读取器，解析 `data:` 行

**容错机制**
- 超时自动重试：reasoning effort 从 high → medium → low 逐级降级
- token 参数协商：如果上游拒绝 `max_output_tokens`，自动切换为 `max_tokens`
- JSON 解析容错：AI 返回的 JSON 经过多层归一化（key 别名映射、类型转换、默认值填充）

### 4.4 判题服务 (Node.js)

约 800 行 JavaScript，ACM/OJ 风格在线判题。

**支持语言与编译参数：**

| 语言 | 编译器 | 标准/版本 | 编译参数 |
|------|--------|----------|---------|
| C | gcc | C11 | `-O2` |
| C++ | g++ | C++17 | `-O2` |
| Java | javac + java | - | `-Xmx256m` |
| C# | dotnet | .NET 8.0 | `build + run` |
| Python | python3 | - | `-u`（unbuffered） |

**判题流程：**
1. 接收代码 + 测试用例 + 语言
2. 写入临时文件到 `/tmp/judge/{uuid}/`
3. 编译（30 秒超时）
4. 逐测试用例运行（5 秒超时，64KB 输出上限）
5. ACM 标准输出比较（去除行尾空白和末尾空行）
6. 生成 checkpoint 摘要（按组聚合通过率和得分）
7. 生成整体判定（AC/WA/CE/RE/TLE/MLE/OLE）
8. 清理临时文件

**测试用例元数据推断：**
- 自动推断 kind：sample（前 1-2 个）、basic（简单）、boundary（边界）、stress（压力）
- 自动生成 checkpoint 标题和分组
- 每个用例附带 feedbackLevel、feedbackTitle、feedbackMessage、nextAction

**AI 代码复审 (judge_review.py)：**
- 触发条件：不是全部通过或得分不是 100
- 4 个评分维度：correctness(0-40)、boundaryRobustness(0-20)、complexityAndPerformance(0-20)、codeQualityAndReadability(0-20)
- 构建结构化 prompt：包含题目上下文、用户代码、语言、判题证据（编译错误/运行时错误/答案错误分类）
- 归一化处理：总分必须等于各维度之和，字符串截断，列表最多 6 项

## 第五部分：安全机制

### 5.1 判题沙箱安全

| 层级 | 措施 | 说明 |
|------|------|------|
| 文件系统 | `read_only: true` | 容器文件系统只读 |
| 临时存储 | `tmpfs /tmp:exec,mode=1777,size=256m` | 仅 tmpfs 可写，限制 256MB |
| 权限 | `cap_drop: ALL` | 丢弃全部 Linux capabilities |
| 提权 | `no-new-privileges: true` | 禁止通过 setuid/setgid 提权 |
| 进程 | `pids_limit: 256` | 限制最大进程数，防止 fork 炸弹 |
| 超时 | 5 秒执行超时 | 防止死循环 |
| 输出 | 64KB 上限 | 超出立即 kill，返回 OLE |
| 编译 | 30 秒编译超时 | 防止编译炸弹 |
| 并发 | 最大 2 个并发任务 | 超出返回 429 |
| 环境 | 沙箱化环境变量 | 仅传递安全的 PATH/HOME/TMPDIR/LANG |
| 认证 | `X-Judge-Token` | 内部 token 认证，防止外部直接访问 |
| 用户 | 非 root 运行 | Dockerfile 创建专用 `judge` 用户 |

### 5.2 认证安全

- 密码哈希：PBKDF2-HMAC-SHA256，120,000 次迭代，16 字节随机盐
- 时序攻击防护：用户不存在时仍执行一次 dummy hash 验证（`resolve_login_password_record`）
- Cookie 安全：httponly、samesite=lax、secure 自动检测（通过 X-Forwarded-Proto）
- 会话 TTL：30 天，过期自动清理
- 输入校验：正则验证用户 ID、邮箱格式、密码长度（6-128）、技能水平和语言白名单

### 5.3 SSRF 防护（文档抓取）

`/api/doc` 端点的 SSRF 防护：
- DNS 解析后检查 IP：拒绝 private/loopback/link-local/multicast/reserved 地址
- 拒绝本地域名：`localhost`、`.local`
- 禁止重定向跟随
- Pinned HTTP 连接：`PinnedHTTPConnection`/`PinnedHTTPSConnection` 防止 DNS rebinding
- 限制：2MB 抓取上限、30,000 字符文本上限、20 秒超时

### 5.4 其他安全措施

- CORS：显式 origin 白名单，可通过 `ALLOWED_ORIGINS` 环境变量配置
- API 密钥隔离：所有 AI API 密钥仅存在于服务器端，前端通过代理转发
- 思维导图限制：每用户最多 200 个，载荷最大 3MB
- 请求体限制：判题服务 1MB
- 部署安全：SSH host key 验证（RejectPolicy），禁止自动接受未知主机

## 第八部分：答辩可能被问到的问题

### 一、架构与技术选型

**Q1: 为什么选择前后端分离而不是 SSR（如 Next.js）？**
A: 本项目核心是交互式可视化动画和客户端状态管理（排序动画、树遍历动画、代码编辑器等），这些场景 CSR 更合适。教程内容相对固定，SEO 需求不强。前后端分离也便于判题服务和 AI 代理的独立部署和扩展。如果未来有 SEO 需求，可以对首页和教程页做 SSG 预渲染。

**Q2: 为什么用 SQLite 而不是 MySQL/PostgreSQL？**
A: 项目部署在单台 VPS 上，用户量级在百人以内，SQLite 的零配置、零运维、文件级备份的优势远大于关系型数据库的扩展性优势。SQLite 在读多写少的场景下性能足够，通过 `threading.Lock` 保证并发写入安全。如果未来用户量增长，可以平滑迁移到 PostgreSQL。

**Q3: 为什么判题服务用 Node.js 而不是用 Go 或 Rust？**
A: 判题服务的核心逻辑是调用系统编译器（gcc/g++/javac）和执行子进程，语言本身不是性能瓶颈。Node.js 的 `child_process` 模块足够胜任，开发效率高，且与前端技术栈统一，降低维护成本。真正的安全隔离靠 Docker 容器层面实现。

**Q4: 为什么用 FastAPI 而不是 Express 统一后端？**
A: FastAPI 的异步特性非常适合 AI API 代理场景——大量请求需要等待第三方 AI API 响应（几秒到几十秒），异步 I/O 可以高效处理并发。同时 Python 生态在 AI/NLP 领域更成熟。判题服务用 Node.js 是因为需要直接管理子进程的 stdin/stdout/stderr，Node.js 的 `spawn` API 更直观。

**Q5: 为什么选择 Tailwind CSS 而不是 CSS Modules 或 styled-components？**
A: Tailwind 的原子化类名在快速迭代 UI 时效率很高，不需要为每个组件想类名。配合 Vite 的 JIT 编译，生产构建只包含实际使用的样式，体积很小。自定义色阶（klein/pine）通过 `tailwind.config.js` 扩展，保持了设计系统的一致性。

**Q6: React 19 相比 React 18 用到了哪些新特性？**
A: 主要用到了 `startTransition` 优化练习页面的筛选交互（避免输入卡顿）、`useDeferredValue` 延迟搜索结果渲染。React 19 的自动批处理也减少了不必要的重渲染。

### 二、功能实现

**Q7: 算法可视化的动画是怎么实现的？**
A: 以排序为例，每种排序算法会预先生成一个完整的动画步骤数组，每一步记录当前数组状态和每个元素的状态（comparing/swapping/sorted/pivot/merge）。播放时通过 `setInterval` 或手动步进逐帧更新状态，React 根据状态渲染对应的柱状图颜色。代码高亮通过每步记录当前执行的代码行号，右侧代码面板据此高亮对应行。这种"预计算+回放"的模式比实时计算更可控，支持暂停、单步、速度调节。

**Q8: 在线判题的输出比较规则是什么？**
A: 采用 ACM/OJ 标准比较规则：去除每行末尾空白字符，去除输出末尾的空行，然后逐行精确比较。这样用户输出末尾多一个换行或空格不会导致 Wrong Answer。

**Q9: AI 出题如何保证生成的题目格式正确？**
A: 三层保障：
1. Prompt 模板严格约束输出格式（JSON schema），包含字段名、类型、示例
2. 后端对 AI 返回的 JSON 做多层归一化：key 别名映射（如 `c++`→`cpp`、`python3`→`python`）、类型转换、缺失字段填充默认值、数组长度校验
3. 前端渲染前再做一次字段存在性检查，缺失的字段优雅降级

**Q10: Vibe Coding 的评分是 AI 打的，如何保证评分的一致性和公平性？**
A: 评分 prompt 中明确定义了四个维度的评分标准和分值范围。后端对 AI 返回的评分做归一化处理：总分必须等于各维度之和，分数 clamp 到有效范围，字符串截断防止过长。虽然 AI 评分存在一定主观性，但通过结构化 prompt 和后处理，评分的一致性在可接受范围内。系统记录历史成绩，用户可以看到自己的进步趋势，这比绝对分数更有参考价值。

**Q11: 思维导图的布局算法是怎么实现的？**
A: 自定义的树形布局算法：
1. 递归计算每个子树的高度（叶节点高度 = 节点高度 + 间距，非叶节点高度 = 子节点高度之和）
2. 从根节点开始，按层级分配 x 坐标（每层固定间距），按子树高度分配 y 坐标（居中对齐）
3. 节点间用贝塞尔曲线连线
4. 支持 CJK 字符的文本换行计算（按字符宽度估算行数）

**Q12: 国际化是怎么实现的？为什么不用 i18next？**
A: 自研了一套轻量级 i18n 方案。原因是项目的中文内容是原生的（不需要翻译），只需要在切换到英文时替换。实现了三层翻译策略：精确匹配 → 正则模式匹配 → 片段替换（最长子串优先）。还有一个 `LocalizationBridge` 组件用 `MutationObserver` 监听 DOM 变化，自动翻译动态插入的文本。这种方案比 i18next 更轻量，且不需要为每个中文字符串都定义 key。

**Q13: 前端 Live Build 模式的 iframe 沙箱是怎么做的？**
A: AI 生成的 HTML/CSS/JS 被合并为一个完整的 HTML 文档（`mergedHtml`），通过 `srcdoc` 属性注入到 `<iframe sandbox>` 中。iframe 的 sandbox 属性限制了脚本的能力（不能访问父页面、不能导航、不能弹窗等），实现了安全隔离。用户可以下载产物为独立的 HTML 文件。

### 三、安全相关

**Q14: 判题服务如何防止恶意代码？**
A: 多层防护：
- Docker 层：只读文件系统、drop ALL capabilities、no-new-privileges、PID 限制 256
- 应用层：5 秒执行超时（kill -9）、64KB 输出上限（超出立即 kill）、30 秒编译超时
- 环境层：沙箱化环境变量（仅 PATH/HOME/TMPDIR/LANG）、非 root 用户运行
- 网络层：内部 token 认证、CORS 白名单
- 并发层：最大 2 个并发任务，超出返回 429

具体来说，fork 炸弹会被 PID 限制拦截；无限循环会被超时 kill；大量输出会被 OLE 限制；文件系统攻击会被 read_only 阻止。

**Q15: 如何防止 SSRF 攻击？**
A: `/api/doc` 端点（用于从 URL 生成思维导图）做了完整的 SSRF 防护：
1. DNS 解析后检查 IP 地址，拒绝 private（10.x、172.16.x、192.168.x）、loopback（127.x）、link-local（169.254.x）、multicast、reserved 地址
2. 拒绝本地域名（localhost、.local）
3. 使用 Pinned HTTP 连接防止 DNS rebinding（解析后固定 IP，不再重新解析）
4. 禁止 HTTP 重定向跟随
5. 限制抓取大小（2MB）和超时（20 秒）

**Q16: 密码是怎么存储的？**
A: PBKDF2-HMAC-SHA256，120,000 次迭代，16 字节随机盐。每个用户的盐不同，存储在数据库的 `password_salt` 字段。登录时用相同的盐和迭代次数重新计算哈希并比较。为防止时序攻击，当用户不存在时仍执行一次 dummy hash 验证。

**Q17: API 密钥是怎么管理的？**
A: 所有 AI API 密钥仅存储在服务器端的 `deploy/.env` 文件中，通过环境变量注入到 FastAPI 容器。前端不存储也不传输 API 密钥，所有 AI 请求通过后端代理转发。部署脚本会将 `LINGMA_JUDGE_INTERNAL_TOKEN` 同步到服务器，禁止使用默认回退 token。

### 四、性能与扩展

**Q18: 项目的并发能力如何？瓶颈在哪里？**
A:
- 前端：Nginx 托管静态资源，并发能力取决于服务器带宽和连接数，通常不是瓶颈
- AI 代理：FastAPI 异步转发到第三方 API，不占用本地计算资源，并发能力取决于第三方 API 的 rate limit
- 判题服务：计算密集型瓶颈，当前限制最大 2 个并发任务。可通过以下方式扩展：
  - 增加 `JUDGE_MAX_CONCURRENT_JOBS`（受 CPU 核心数限制）
  - Docker Compose 中 `scale` 判题容器
  - 引入消息队列（如 Redis）做任务调度

**Q19: SQLite 的并发写入怎么处理？**
A: 通过 Python 的 `threading.Lock`（`db_lock`）全局锁保证同一时刻只有一个线程写入 SQLite。这在当前用户量级下足够，但如果并发写入成为瓶颈，可以：
1. 启用 SQLite WAL 模式（允许并发读写）
2. 迁移到 PostgreSQL

**Q20: 前端首屏加载性能如何优化？**
A:
- 路由级代码分割：首屏只加载 Home 页面的代码
- Vite 构建优化：tree-shaking、代码压缩、CSS 提取
- Nginx gzip 压缩
- 骨架屏避免白屏感知
- 翻译资源按需加载（中文模式下不加载英文翻译）
- 题库数据并行懒加载（进入 Practice 页面才加载）

### 五、项目管理与工程化

**Q21: 项目的代码规模有多大？**
A:
- 前端：约 80 个 TypeScript 源文件，涵盖 10 个页面、5 个可视化组件、9 个教程组件、7 个服务模块、3 个 Context、3 个自定义 Hook、12 个 i18n 模块
- 后端 API 代理：约 3400 行 Python（main.py）+ 约 300 行（judge_review.py）
- 判题服务：约 800 行 JavaScript
- 部署配置：Docker Compose + 3 个 Dockerfile + Nginx 配置 + 自动化部署脚本
- 数据文件：6 章课程内容、7 个题库模块、10 个方法论单元

**Q22: 构建验证流程是什么？**
A: 部署前必须通过：
1. `npx tsc -b` — TypeScript 类型检查（与服务器一致的严格模式，会报 unused imports）
2. `npx vite build` — 生产构建
3. `vitest run` — 单元测试

**Q23: 为什么用 paramiko 而不是直接 SSH 部署？**
A: 开发环境是 Windows，没有原生的 `ssh`/`scp`/`sshpass` 命令。paramiko 是纯 Python 的 SSH 实现，跨平台兼容。部署脚本还做了 SSH host key 验证（RejectPolicy），不会自动接受未知主机。

### 六、设计与用户体验

**Q24: 为什么要做 Vibe Coding 模块？和数据结构学习有什么关系？**
A: Vibe Coding 模块的定位是"提示词工程训练"，它和数据结构学习的关系是：在 AI 时代，能否写出高质量的提示词来让 AI 辅助编程，已经成为一项重要的工程能力。Vibe Coding 模块让学生在学习数据结构的同时，也训练与 AI 协作的能力。5 个赛道（前端/后端/调试/重构/审查）覆盖了软件开发的主要场景，4 个评分维度（目标清晰度/边界约束/验证设计/输出格式）是提示词质量的核心要素。

**Q25: 深色模式为什么按时间自动切换？**
A: 研究表明，夜间使用深色界面可以减少眼睛疲劳。自动切换（19:30-7:30 深色，7:30-19:30 浅色）让用户无需手动操作就能获得舒适的视觉体验。同时保留了手动切换的选项，手动切换后会禁用自动模式。每 60 秒检查一次时间，确保跨时段使用时能自动切换。

**Q26: 低动效模式是怎么检测的？**
A: 通过 `useLowMotionMode` hook 检测两个条件：
1. `prefers-reduced-motion: reduce` — 用户在系统设置中开启了减少动画
2. 移动设备检测 — `max-width: 768px` 或 `hover: none` 或 `pointer: coarse`
满足任一条件即启用低动效模式，所有 Framer Motion 动画退化为瞬时切换或静态渲染。

**Q27: 个性化推荐算法是怎么工作的？**
A: `buildDailyRecommendation()` 算法：
1. 根据最近练习表现调整有效技能水平（连续做对 → 升级，连续做错 → 降级）
2. 判断学习状态：
   - onboarding：新用户，推荐入门题
   - review：有错题需复习，优先推荐错题
   - steady：稳定练习，按路径推进
   - challenge：表现优秀，推荐更难的题
3. 基于日期哈希确定性选择每日焦点分类（同一天同一用户看到相同推荐）
4. 在焦点分类中选择未完成的、难度匹配的题目

### 七、对比与改进

**Q28: 和 LeetCode、力扣等平台相比，灵码的优势和不足是什么？**
A:
优势：
- 算法可视化：LeetCode 没有交互式动画演示，灵码可以逐步看到算法执行过程
- 结构化教程：按教材章节组织，适合系统学习，而非刷题
- AI 智能出题：根据用户水平动态生成题目，不局限于固定题库
- Vibe Coding 训练：提示词工程训练是独特功能
- 轻量部署：单台 VPS 即可运行

不足：
- 题库规模远小于 LeetCode（百题级 vs 万题级）
- 没有社区讨论、排行榜等社交功能
- 判题沙箱安全性不如专业 OJ（没有 cgroup 级别的资源隔离）
- 不支持更多编程语言（如 Go、Rust）

**Q29: 如果要支持更多用户，需要做哪些改进？**
A:
1. 数据库：SQLite → PostgreSQL，支持并发写入
2. 判题服务：引入消息队列（Redis/RabbitMQ），水平扩展判题 worker
3. 缓存：Redis 缓存热点数据（用户会话、题库数据）
4. CDN：静态资源上 CDN，减少服务器带宽压力
5. 监控：接入 Prometheus + Grafana，监控服务健康和性能指标
6. 判题安全：引入 nsjail 或 isolate 做进程级沙箱隔离

**Q30: 项目中遇到的最大技术挑战是什么？**
A:（建议根据实际经历回答，以下是参考）
1. AI 返回格式不稳定：不同模型返回的 JSON 格式差异很大，需要做大量的归一化处理和容错
2. SSE 流式传输的全链路调通：从 AI API → FastAPI → Nginx → 前端，每一层都需要正确配置（Nginx 关闭 buffering、FastAPI 的 StreamingResponse、前端的 EventSource 解析）
3. 判题服务的安全隔离：需要在允许编译执行用户代码的同时防止恶意行为，Docker 层面的安全配置需要仔细调优
4. 可视化动画的性能：排序动画在大数据量时需要优化渲染性能，避免频繁的 DOM 更新

### 八、理论基础

**Q31: 项目用到了哪些设计模式？**
A:
- 代理模式：FastAPI 作为 AI API 和判题服务的代理
- 观察者模式：React Context 的 Provider/Consumer、MutationObserver 监听 DOM 变化
- 策略模式：多种 AI 供应商的适配、多种排序算法的可视化
- 工厂模式：根据算法 ID 创建对应的可视化组件和教程组件
- 单例模式：SQLite 数据库连接、全局锁

**Q32: 项目的前端架构属于什么模式？**
A: 典型的 SPA（Single Page Application）架构，使用 React Router 做客户端路由。状态管理采用 React Context + localStorage 的组合，没有引入 Redux 等重量级状态管理库，因为项目的全局状态较简单（用户信息、主题、语言）。组件按功能分层：pages（页面）→ components（UI 组件）→ services（API 调用）→ contexts（全局状态）→ hooks（可复用逻辑）→ utils（工具函数）→ data（静态数据）。

**Q33: RESTful API 的设计原则在项目中是怎么体现的？**
A:
- 资源命名：`/api/auth/session`、`/api/mindmaps`、`/api/vibe-coding/history`
- HTTP 方法语义：GET 获取资源、POST 创建/操作资源
- 状态码：200 成功、400 参数错误、401 未认证、403 禁止、404 未找到、409 冲突、429 限流、500 服务器错误、502 上游错误、503 服务不可用、504 超时
- 不过也有不完全 RESTful 的地方：部分端点用 POST 做查询（如 `/api/mindmaps/load`），这是为了在请求体中传递用户认证信息

**Q34: Docker 容器化的优势在这个项目中体现在哪里？**
A:
1. 环境一致性：判题服务需要 gcc/g++/javac/python3/.NET，Docker 保证了开发和生产环境完全一致
2. 安全隔离：判题容器的 read_only、cap_drop、pids_limit 等安全配置只有容器化才能方便实现
3. 一键部署：`docker-compose up -d` 启动全部三个服务，无需手动安装依赖
4. 服务隔离：三个服务独立容器，互不影响，可以独立重启和扩展
5. 资源管理：可以为每个容器设置 CPU/内存限制
